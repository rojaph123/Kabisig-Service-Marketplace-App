# Kabisig QA Report

Date: 2026-04-25  
Reviewer: Codex  
Scope: `mobile-app`, `admin-web`, `shared`, Firebase rules/config  
Project mode assessed: free-plan / no deployed Cloud Functions

## Executive Summary

This system is a strong MVP and is usable for continued testing and demo work, but it is **not ready for final production sign-off**.

The main reasons:
- security/rules still have important gaps
- admin and mobile data-access patterns are not fully aligned
- several features are functionally present but not fully verified end-to-end
- dark mode and web UX warnings still need a dedicated cleanup pass

## Current Critical Issues

### 1. `bookingSlots` rules are too permissive

Status: **Open**

Problem:
- any signed-in user can currently create, update, or delete booking slot lock documents

Impact:
- booking integrity can be tampered with
- malicious or accidental slot blocking is possible
- booking conflict logic cannot be trusted fully under hostile conditions

Reference:
- [firestore.rules](/C:/Users/Danielle%20Blanca/Videos/kabisig/firestore.rules:89)

Required fix:
- restrict write access so normal users cannot arbitrarily modify any slot document
- if staying on free plan, the safest realistic approach is to constrain slot ownership and update/delete behavior by document fields

### 2. Notifications can still be spoofed

Status: **Open**

Problem:
- any signed-in user can create notifications for any existing user

Impact:
- fake unread messages
- fake booking status notifications
- fake payment notifications

Reference:
- [firestore.rules](/C:/Users/Danielle%20Blanca/Videos/kabisig/firestore.rules:156)

Required fix:
- restrict notification creation much harder
- only allow:
  - admin-created notifications
  - or tightly scoped self-originating cases that are enforceable in rules

### 3. Provider approval workflow is still contradicted by customer-side fallback logic

Status: **Open**

Problem:
- if there are no approved providers, the app intentionally falls back to showing non-rejected, non-draft providers

Impact:
- customers can see providers who are not actually approved
- approval workflow credibility is weakened

References:
- [shared/src/firebaseServices.ts](/C:/Users/Danielle%20Blanca/Videos/kabisig/shared/src/firebaseServices.ts:1076)
- [shared/src/firebaseServices.ts](/C:/Users/Danielle%20Blanca/Videos/kabisig/shared/src/firebaseServices.ts:1108)

Required fix:
- remove fallback visibility for non-approved providers
- if you need test data, create approved test providers instead

## High Priority Issues

### 4. Admin provider approval filters are not fully truthful

Status: **Open**

Problem:
- the approvals page shows filters for `Revision Requested`, `Rejected`, and `Approved`
- but the snapshot only keeps `Pending Approval` applications

Impact:
- admin UI suggests filtering breadth that does not actually exist

References:
- [admin-web/lib/marketplace-data.ts](/C:/Users/Danielle%20Blanca/Videos/kabisig/admin-web/lib/marketplace-data.ts:236)
- [admin-web/app/(dashboard)/provider-approvals/page.tsx](/C:/Users/Danielle%20Blanca/Videos/kabisig/admin-web/app/(dashboard)/provider-approvals/page.tsx:15)

Required fix:
- either load all application statuses into admin snapshot
- or remove unsupported filter options

### 5. Booking conflict history writes are also too open

Status: **Open**

Problem:
- any signed-in user can create `bookingConflicts` documents

Impact:
- conflict history can be polluted
- admin reporting can be falsified

Reference:
- [firestore.rules](/C:/Users/Danielle%20Blanca/Videos/kabisig/firestore.rules:83)

Required fix:
- reduce write permissions
- ideally only allow conflict writes in a tightly constrained booking path

### 6. Mobile identity lookups are improved but still structurally weak

Status: **Partially fixed**

What was fixed:
- removed full `users` collection reads in key screens

Remaining concern:
- the app still depends on reading user docs for public display data
- `/users` now allows any signed-in user to read any user document, which is convenient but broad

Impact:
- permission errors are reduced
- privacy boundary is weaker than ideal

References:
- [firestore.rules](/C:/Users/Danielle%20Blanca/Videos/kabisig/firestore.rules:48)
- [shared/src/firebaseServices.ts](/C:/Users/Danielle%20Blanca/Videos/kabisig/shared/src/firebaseServices.ts:652)

Recommended improvement:
- move public display fields into a dedicated public profile shape
- keep sensitive user fields out of broad-read docs

## Medium Priority Issues

### 7. Dark mode is still not complete

Status: **Open**

Observed risk areas:

- some hero/header shells
- some shared card/input/modal surfaces
- chat and profile details likely still have isolated low-contrast states

Impact:

Suggested next step:

### 8. Console warnings still exist on web

Status: **Partially fixed**

Likely harmless:
- `Download the React DevTools...`
- `Running application "main"...`

Still needing cleanup:
- `shadow* style props are deprecated. Use "boxShadow".`
- `aria-hidden` focus warnings
- touch responder noise in web

Impact:
- does not necessarily break flows
- does indicate incomplete web polish/accessibility cleanup

### 9. Foreground notifications are acceptable for free-plan MVP, but limited

Status: **Accepted limitation**

Current behavior:
- in-app realtime notifications
- foreground/local notification path only

Not supported:
- phone-off push
- background remote delivery

Assessment:
- acceptable only if your documentation and expectations are explicit

## Suggestions

### Product / Technical Suggestions

1. Stop adding major features until the current flows are stabilized.
2. Treat the current app as an MVP that needs a structured stabilization sprint.
3. Seed realistic test accounts and test data:
   - customer
   - provider pending
   - provider revision requested
   - provider approved
   - provider rejected
   - admin
4. Create one QA sheet per role and execute it systematically.
5. If you stay on free plan, document the limits clearly:
   - no true background push
   - no deployed Cloud Functions
   - client-enforced admin actions with rules as protection
